## data-excel.mjs v2 优化（转换脚本本身，不动数据）

### 背景
用户希望优化 `scripts/data-excel.mjs`（Excel↔TS 双向转换工具）本身，不动数据。
现有工具链：`转Excel.bat` / `转回Ts.bat` 调用 `node scripts/data-excel.mjs export|import`。

### 优化点（已实施）
1. **`lit()` 修复对象 key 引号 bug**：旧版把对象 key 直接拼成 `${k}:`，遇到 `special-attack` / `special-defense` 等带连字符的 key 会生成非法 TS（已确认 `pokemon-mega.ts` 里就有 `special-attack: 122` 这种语法错误，被 Vite 容错过去但 esbuild 严格解析会报错）。新增 `keyLit(k)` 辅助函数：合法标识符/纯整数不加引号，其余用 `JSON.stringify`。
2. **`lit()` 支持嵌套缩进**：加 `indent` 参数，递归时递增，深层结构可读性更好。
3. **`loadTS()` 加文件级缓存**：用 `Map<file, Module>` 缓存，同次运行内只跑一次 esbuild.transform + import。tools.ts 三张表（小工具/属性相克/防御矩阵）、复合表的 movesFile、外键 lookup 不再重复加载。
4. **import 时 EBUSY 友好提示**：xlsx 被 Excel 占用时给出明确提示而非原始堆栈。
5. **告警汇总**：全局 `warnCount` 收集 `parseLevelMoves`/`parseNameList` 的警告，末尾打印总数。
6. **跳过空行**：`readRows` 跳过第一列为空的行（避免 Excel 末尾空行被当成数据）。**关键**：用第一列判断而非 `field.kind==='id'`，因为宝可梦"编号"的 kind 是 'num'、小工具"id"的 kind 是 'str'，不是 'id'。
7. **`--dry-run` 选项**：`import [--dry-run]` 只打印将写哪些文件+行数，不实际写回。
8. **总览日志**：export/import 末尾打印总表数/总行数/耗时。

## 修复 app一键打包.bat 编码

### 第一次修复：GBK/ANSI
用户双击运行新脚本后，CMD 窗口出现 `不是内部或外部命令` 等报错，且最终提示"未找到 managed Node"。
原因：脚本被写为 UTF-8 without BOM，而中文 Windows 的 CMD 默认使用 GBK（code page 936）读取批处理文件；UTF-8 多字节中文被 GBK 拆开后，某些字节被 CMD 当作命令/特殊字符，导致 `echo` 后面的中文被解析成独立命令。
修复：用 Python 将 `app一键打包.bat` 从 UTF-8 转码为 GBK（ANSI）保存，`file` 显示为 "Non-ISO extended-ASCII, with CRLF line terminators"。

### 第二次修复：UTF-8 + chcp 65001（最终方案）
转 GBK 后，脚本逻辑能跑（Node/JDK 都识别），但中文输出全部显示为 `?`（说明用户 CMD 当前代码页不是 GBK，可能是 UTF-8 或其他）。
原因：GBK 文件内容被非 GBK 代码页读取，导致中文乱码为 `?`。
最终修复：把脚本改回 **UTF-8 without BOM**，并在文件最开头加 `chcp 65001 >nul`，随后才是 `@echo off` / `setlocal`。这样 UTF-8 中文输出被强制按 UTF-8 解析，不依赖系统默认代码页，兼容 GBK 与 UTF-8 控制台。统一换行符为 CRLF。

### 改动文件
- `app一键打包.bat`（最终：UTF-8 + chcp 65001 + 标准 CRLF）
